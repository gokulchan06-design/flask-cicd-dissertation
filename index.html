A controlled experiment comparing two AWS CI/CD pipeline configurations across 40 deployment runs and six DORA metrics.

The Problem

Small teams debate pipeline complexity every day — without evidence

Teams building on AWS must choose between a simple, fast pipeline that deploys every commit directly, or a more complex pipeline with testing, security scanning, and human approval. Most make this decision based on opinion. This dissertation provides empirical evidence.

Primary Research Question
How do different AWS CI/CD pipeline configurations affect deployment reliability, failure recovery time, and service availability for a small containerised web application?
SRQ1 Does automated unit testing in the pipeline reduce the deployment failure rate?
SRQ2 Does a manual approval gate increase lead time, and by how much?
SRQ3 What is the cost-reliability trade-off when increasing pipeline complexity?
SRQ4 Does container security scanning detect real vulnerabilities before deployment?

Answers to Research Questions

What the data actually shows

Direct empirical answers to the four secondary research questions, based on 40 controlled AWS experiments.

🔬
SRQ1: Does testing reduce failure rate?
Yes — definitively. The pytest Test stage intercepted all 4 flawed commits within a mean of 44 seconds, before any Docker image was built. The baseline pipeline deployed all 4 bugs silently while reporting pipeline status as Succeeded.
20% → 0% bug deployment rate · 100% reduction
⏱
SRQ2: Does approval increase lead time?
Yes. Mean lead time increased from 243s to 387s — a 59.4% increase. The Manual Approval stage accounted for ~90 seconds of this. Both pipelines remain within the DORA elite benchmark of under 1 hour.
+144.3 seconds (+59.4%) · still DORA elite tier
💰
SRQ3: What is the cost trade-off?
The trade-off strongly favours the enhanced pipeline. A 100% reduction in production defects and complete elimination of rollback incidents was achieved at $0.00 additional cost. Both pipelines ran entirely within the AWS free tier.
$0.00 additional cost · best possible trade-off
🔒
SRQ4: Does security scanning detect CVEs?
Yes. Trivy identified real CVE vulnerabilities in the python:3.11-slim base image on each enhanced pipeline run. These would have been silently deployed by the baseline pipeline. Scanner executed in ~15-20 seconds per build.
Real CVEs detected · would have been silently deployed
Answer to Primary Research Question
The enhanced pipeline significantly improves deployment reliability: bug deployment rate reduced by 100%, rollback incidents eliminated entirely, at identical service availability (99.88%) and zero additional AWS cost. Lead time increased by 59.4% — a modest trade-off for complete elimination of production defects.
Research Contributions

Why this research matters

This dissertation makes three distinct contributions to cloud computing and software engineering research — filling a gap that has existed since the DORA framework was established in 2018.

1
First controlled experiment at small scale on AWS
No previous study combines a controlled experimental design, single-developer context, AWS-only infrastructure, and measurement of all six DORA-aligned metrics including cost. Most CI/CD research uses surveys of large enterprises. This study isolates the causal effect of pipeline design through experimental control — enabling stronger conclusions than surveys allow.
2
Reproducible reference implementation on GitHub
The complete system — Flask application, Dockerfiles, buildspec configurations, IAM role definitions, metric collection scripts, and availability monitoring tool — is publicly available and fully reproducible. Future practitioners and researchers can adopt, adapt, or extend this implementation without starting from scratch.
3
Evidence-based recommendation for small AWS teams
The evidence is unambiguous: the enhanced pipeline eliminates production defects at zero additional financial cost with only 2.5 minutes of additional deployment time. For any small team where application correctness matters — which is essentially all production deployments — there is no empirical justification for using a pipeline without automated testing.
